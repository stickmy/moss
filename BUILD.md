## Building

### Moss App

```bash
xcodegen generate
xcodebuild -project Moss.xcodeproj -scheme Moss -configuration Debug build
```

The app binary lands in `~/Library/Developer/Xcode/DerivedData/Moss-*/Build/Products/Debug/Moss.app`.

To run:

```bash
open ~/Library/Developer/Xcode/DerivedData/Moss-*/Build/Products/Debug/Moss.app
```

### Moss CLI

The `moss` CLI has no dependency on GhosttyKit and can be built directly with SPM:

```bash
swift build --target MossCLI
```

Binary output: `.build/debug/moss`

### GhosttyKit (vendored libghostty)

GhosttyKit is a vendored xcframework containing `libghostty.a` (the Ghostty terminal C library) and `ghostty.h`. It is checked into the repo and only needs rebuilding when updating to a newer Ghostty version.

```bash
# Clone ghostty source (one-time)
git clone https://github.com/ghostty-org/ghostty.git /path/to/ghostty

# Build and update vendored files
./Vendor/build-ghostty.sh /path/to/ghostty
```

What the script does:

1. Fetches zig package dependencies (`zig build --fetch`)
2. Patches upstream `build.zig` to install `libghostty.a` on Darwin (upstream only does this on Linux/Windows)
3. Builds arm64 and x86_64, then `lipo` into a universal binary
4. Strips debug symbols (267MB â†’ ~45MB)
5. Copies `libghostty.a` and `ghostty.h` into the xcframework
6. Reverts the patch so the ghostty checkout stays clean

The `module.modulemap` in the xcframework renames the C module from `libghostty` to `GhosttyKit`, so Swift code uses `import GhosttyKit`. **Do not overwrite it** â€” only `libghostty.a` and `ghostty.h` should be updated.

## Dependencies

### GhosttyKit

Vendored locally at `Vendor/GhosttyKit.xcframework/`. Linked as a static library (not embedded in the app bundle). Requires `libc++` and `Carbon.framework` at link time, both declared in `project.yml`.

### Tree-sitter Grammars (17 SPM packages)

Used for syntax highlighting in the file preview panel. Most track `master`/`main`, but **four are pinned to specific revisions** because their `Package.swift` uses `FileManager.default.fileExists(atPath: "src/scanner.c")` to conditionally compile the scanner â€” a check that only works when SPM's working directory is the package root. Xcode sets a different directory, so the check silently fails and causes `Undefined symbol: _tree_sitter_xxx_external_scanner_create` at link time.

Pinned packages (do not change to `branch: master`):

| Package | Pinned Revision |
|---------|----------------|
| tree-sitter-python | `53639fb` |
| tree-sitter-javascript | `a48cee8` |
| tree-sitter-css | `6e4e788` |
| tree-sitter-yaml | `ccfaef1` |

Before updating any tree-sitter grammar, verify the new revision's `Package.swift` does NOT use `fileExists` for scanner detection.

## Xcode Project Generation

The `.xcodeproj` is generated by XcodeGen from `project.yml` and is gitignored. Regenerate after:

- Changing `project.yml`
- Adding or removing source files
- Adding or removing SPM packages

```bash
xcodegen generate
```

**YAML trap**: `project.yml` has a single `dependencies:` key under the Moss target containing SPM packages, the CLI target, the vendored xcframework, and SDK frameworks. Never add a second `dependencies:` key â€” YAML silently overwrites the first, dropping all entries.

## Troubleshooting

### SPM "Unable to find module dependency"

Package cache is corrupted. Clean and regenerate:

```bash
rm -rf Moss.xcodeproj
rm -rf ~/Library/Developer/Xcode/DerivedData/Moss-*
xcodegen generate
open Moss.xcodeproj
# Wait for "Resolving Package Graph..." â†’ âŒ˜B
```

### GhosttyKit Build Failures

| Error | Cause | Fix |
|-------|-------|-----|
| `cannot execute tool 'metal'` | Missing Metal Toolchain | `xcodebuild -downloadComponent MetalToolchain` (~700MB, one-time) |
| `unable to connect to server` | zig HTTP fetch failure | Run `cd /path/to/ghostty && zig build --fetch` first |
| `invalid option: -Dinspector` | Upstream removed the flag | Update build script to remove it |
| `libghostty.a` not found | Used `-Dapp-runtime=none` | That only builds libghostty-vt (parser only), not the full embedded runtime |

### Linker Errors

| Symbol | Framework |
|--------|-----------|
| `_kTISPropertyUnicodeKeyLayoutData`, `_TISCopyCurrentKeyboardLayoutInputSource` | `Carbon.framework` |
| `_tree_sitter_xxx_external_scanner_create` | Tree-sitter grammar pinning issue (see above) |

### Codesign "code object is not signed at all"

`libghostty.a` is being copied into `Moss.app/Contents/Frameworks/`. Ensure `project.yml` has `embed: false`:

```yaml
- framework: Vendor/GhosttyKit.xcframework
  embed: false
```
